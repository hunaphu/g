Explanation

I thought a UTF-16(LE) based approach might be more efficient for Chinese characters. The Traditional Chinese table's UTF-16 code units have 16 distinct leading bytes, so I compressed each code unit to 1.5 bytes (3 hex digits), using a lookup table. Crucially, 0x30 is mapped to 0 so that the ideographic spaces are just consecutive zeroes for the base85 encoding + string format.

The lookup table is tacked onto the end of the hex string, and a regular expression does the substitutions. The lookup table's entries are separated by 66, this is simply a two-digit hex string that does not otherwise appear in the encoded text.

Possible areas for improvement

  • I initially thought I could get away with no environment set-up bytes:
    • The sqlite3 CLI documentation led me to believe that on Windows, UTF-16 would be the default console output, but I was not personally able to get that to work, so the PRAGMA was necessary.
    • The creator of the sqlean extensions also provides a pre-loaded sqlite3 + sqlean shell, but this doesn't appear to support PRAGMA encoding='utf16'. Having tried a few things, this feels buggy, as it seems to be the loading of the sqlean extension bundle which causes the issue, but it works to set the encoding before load.
  • I tried a Python version, but the functions were too wordy. However, I was able to append |.* to the regex and for reasons I don't fully understand, that also did the substring to remove the lookup table. It did not do so for this regexp extension, though perhaps a similar change to the regex might be less bytes than the left function.
  • Based on all the environment set-up bytes, and regexp_replace not being that short, I suspect a lower byte-count might be achievable by taking the sqlite output after the hex call and piping it to some perl regex shenanigans, but my brain is dead at this point, and a pure sqlite solution seems like a good stopping point for me.

Encoding

For each element name, the Unicode code point is broken down into:

  • a 2-byte prefix
  • a 3rd byte in the range \$[128\dots191]\$
  • a 4th byte in the same range if the prefix starts with 0xF0

There are 15 distinct prefixes.

Hence the following encoding on 10 bits:

vvvvvvpppp
\____/\__/
   |    |
   |    '--> prefix identifier
   '-------> value of 3rd byte, minus 128

The upper 8 bits and the lower 2 bits of these 10-bit values are packed into two separate Base64 streams.

Special values lower than 9 are used for line-feeds and repeated whitespace.

If the prefix identifier is either 0 or 1, a 4th byte is taken from a separate lookup table.
