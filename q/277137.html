Revised answer based on criteria without reflectional symmetry. If the input corresponds to a set of four 1 bits on the faces of the dodecahedron arranged in a Z pattern (or part of such a set) returns false. Otherwise returns true.

The Z pattern has 2 fold rotational symmetry, therefore there are only 30 possible orientations of this pattern on the dodecahedron, and hence 30 possible sets of four 1 bits the input that return falsy. If we were to test for each oriantation separately, then for each orientation of the Z pattern, there would be a 2-fold rotation of the inputs that leaves the output unchanged. Hence there are 60 possible rotations of the inputs that leave the output unchanged.

As a result of golfing, false is also returned for the following additional cases bringing the total to 229: all inputs with 0,1,or 2 1 bits (1+12+66=79); inputs with 3 1 bits corresponding to part of a Z with one of the four bits deleted (4*30=120).

Explanation

The face labelling system used in this answer is from my answer to this Steiner system question. Note that opposite faces always have indices 6 apart, and the labelling has six fold symmetry in that offsetting indices by 2 corresponds to rotating the dodecahedron 1/6 turn and flipping at the equator. Because we don't want to detect the Z shape but not its reflection (the S shape) we only consider offsetting indices by 4, which corresponds to a 1/3 rotation.

The ends of the Z shape are on opposite faces of the dodecahedron. They are therefore either both on equatorial faces or both on polar faces. These are listed below - one example for each pair of opposite faces. Note that because the opposite faces are pentagons, there are 5 possible rotations per pair of opposite faces, generating 30 possible orientations of the Z shape.

Equatorial  0,6 with 2,4    4,10 with 6,8       8,2 with 10,0
Polar       3,9 with 4,6    7,1, with 8,10     11,5 with  0,2

The offsets for each possible rotation (5 for equatorial ends and 5 for polar ends) are stored in the magic string. They are decoded from the ascii code in j as j/12 and j%12 (because the order does not matter, careful selection of the encoding allows only printable characters to be used. For even iterations, the expression (65|j<<j/12|1<<1%12) generates the 4 bits for the equatorial case with end faces 0 and 6. 65 in binary is 1000001 and the expressions with j fill in the remaining two bits. For odd iterations a rotated version of the polar case with end faces 11 and 5 is generated.

Variable i keeps track of iterations 0 to 29. The expression above is multiplied by 4097 to duplicate it in bits 12 to 23 to allow for wrapping. For even iterations the result is rightshifted by i%3*4 to simulate a 1/3 rotation. For odd iteration the additional +i%2 imparts an additional shift of 1 (changing faces 0 and 6 to 11 and 5 as mentioned above, for example.)

This generates all 30 possible orientations of the Z shape in bits 0 to 11, in some cases with nonsense in higher bits. One way to clean this up would be to AND with 4095. But it is shorter to AND with the input n. This however means that when the result is compared with n matches are found not only for a complete Z shape but also for fragments of the Z shape, as described above.

->n{z=i=0                     #take an input n. Set up a counter i. Set z to 0, a truthy value in Ruby.
("2%+@kgvv=#"*3).bytes{|j|    #triple the magic string making a string of length 30. Iterate 30 times.
z&&=n!=(                      #if the following expression equal to z, set z to false.
(65|1<<j/12|1<<j%12)          #take 65 representing the ends of the Z
                              #and use the magic string to fill in remaining faces
*4097>>i%3*4+i%2)&n           #make a duplicate in upper bits and shift right. 
                              #AND with n to remove bits beyond bit 11.
i+=1}                         #increment i
z}                            #return z true or false
